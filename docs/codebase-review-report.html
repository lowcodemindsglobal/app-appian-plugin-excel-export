Executive Summary

This repository implements a single Appian Smart Service plug-in: it runs one or more SQL queries against a JNDI-configured data source and writes the combined results into a multi-sheet, column-protected .xlsx workbook, optionally password-encrypted, saved back into Appian as a Document. The codebase is intentionally small — five main classes, two test classes, and Appian's required plug-in descriptor/resource files — and it stays inside that scope without unnecessary abstraction.

The implementation quality is good relative to its size: an immutable, fail-fast configuration object built via the Builder pattern, a streaming (SXSSF) Excel generator validated against a 50,000-row dataset, careful reuse of cell styles to avoid Apache POI's per-workbook style ceiling, and consistent try-with-resources use around JDBC and Appian content APIs. Documentation — both in the README and in code comments — is unusually thorough, recording several hard-won, empirically-verified Appian deployment gotchas (input type mapping limits, i18n bundle path conventions, deprecated-API usage constraints) that would otherwise cost a future maintainer real deployment cycles to rediscover.

The review's most significant findings are concentrated in two areas: (1) the SQL query "validation" in AdvanceExcelExport.validate() is a naive string check (starts with SELECT, doesn't end with ;) that provides only superficial protection against malformed or malicious query text, and (2) AdvanceExcelExport — the class wired directly into the Appian runtime, containing all of the orchestration, validation, and document-overwrite logic — has no unit tests at all; only its collaborators (ExcelExportConfig, ExcelGenerator) are tested. Neither finding is a critical vulnerability or defect in the code as it exists today, but both represent real exposure as the plug-in is adopted more widely across process models.

No critical-severity issues were found. The plug-in is safe to deploy in its current state for use by trusted process designers who do not build SQL query text from untrusted end-user input; the recommendations below should be treated as hardening work for broader or higher-trust-boundary adoption.

Project Strengths

Immutable, fail-fast configuration

ExcelExportConfig is built exclusively through a validating Builder; required fields throw NullPointerException at build time rather than surfacing as null-pointer failures deep inside generation logic.

Genuinely streaming Excel generation

SXSSFWorkbook with a 100-row access window keeps memory flat regardless of row count, and this is backed by a real test (testLargeDataset) generating 50,000 rows, not just asserted in a comment.

Cell-style cache avoids a real POI failure mode

DataStyles pre-creates and reuses 6 style combinations per workbook instead of creating one per cell — sidesteps Apache POI's hard cap on distinct cell styles per workbook, a common defect in hand-rolled exporters.

Real JDBC integration tests, not mocks

ExcelGeneratorTest runs an in-memory H2 database and exercises the actual ResultSetMetaData → cell-value pipeline, catching type-mapping and formatting bugs that hand-mocked ResultSet objects would miss.

Unusually thorough operational documentation

The README and in-code comments record specific, empirically-verified Appian deployment failures (unsupported Map[] input type, i18n bundle path convention, deprecated-API constraints) and why the current approach avoids them — this is hard-won knowledge, not boilerplate.

Consistent resource management

Connection, PreparedStatement, ResultSet, and ContentOutputStream are all acquired via try-with-resources; the single JDBC connection is correctly shared and reused across every sheet in one export run.

Dual failure signaling

run() both populates the ErrorMessage output and throws SmartServiceException, so a calling process branching on the error output and one relying on a fault handler both see the failure — no silent continuation on a null document.

Deliberate, explained dependency policy

maven-enforcer-plugin bans log4j, Netty, Jetty, and JUnit 4 transitively; the POI-shading relocation exception (XMLBeans left unrelocated) is justified with a specific, verified runtime failure mode rather than asserted without evidence.

Critical Findings

None identified. No hardcoded credentials, authentication/authorization bypass, remote code execution, or data-loss vectors were found in the reviewed code.

High Priority Findings

High Security — Injection

SQL query "validation" is a superficial string check, not a real safeguard

AdvanceExcelExport.java — validateSheetData(), lines 184–200

Description
The only checks applied to each sheet's SQL text are: non-blank, case-insensitive starts-with SELECT, and does-not-end-with ;. There is no real SQL parsing, no statement-count check, and no defense against SQL comments, stacked expressions, vendor-specific data-exfiltration constructs (e.g. SELECT ... INTO OUTFILE, OPENROWSET), or a query that calls into a mutating stored function via a SELECT. The query itself is also never parameterized — the plug-in has no bind-variable mechanism, so any caller who needs to filter by a runtime value is forced into string concatenation.
Why it matters
Whether this is exploitable depends entirely on who controls the query text. Today it is presumably authored by trusted process designers at build time, but the plug-in provides no mechanism to keep it that way, and no documentation warns callers against building this string from process-variable or user-supplied data.
Potential impact
If any calling process model ever concatenates user-supplied input into a SQLSheetData.sqlQuery value, this becomes a classic SQL injection vector against whatever data source the JNDI name resolves to — with read access broad enough to run arbitrary SELECT statements, this could expose or exfiltrate data far beyond the intended report scope.
Recommendation
Document explicitly (README + input description) that sqlQuery must never be built from end-user input without independent parameterization/escaping upstream. Consider adding a lightweight SQL statement-count guard (reject text containing a semicolon anywhere, not just at the end) and, if feasible, a real single-statement parse check via the JDBC driver (e.g. Connection.prepareStatement validation combined with a statement-type check) rather than string prefix matching.
High Testing

The orchestration class wired to the Appian runtime has zero test coverage

AdvanceExcelExport.java — entire class (294 lines); no corresponding test file exists

Description
Both test files in the repository (ExcelExportConfigTest, ExcelGeneratorTest) test collaborators. AdvanceExcelExport itself — validate()'s field-level rules, run()'s exception wrapping, and findExistingDocumentId()/saveAsDocument()'s create-vs-overwrite branching — has no automated coverage at all.
Why it matters
This class contains the plug-in's only branching logic around document overwrite detection (getByPath lookup, VERSION_CURRENT vs. UNIQUE_FOR_PARENT upload mode) and its only error-surfacing logic (errorMessage assignment, exception rethrow). Both are exactly the kind of logic that regresses silently during refactors because nothing fails a build when it breaks.
Potential impact
A future change to the overwrite-detection logic (e.g. adjusting findExistingDocumentId) could silently start creating duplicate documents instead of overwriting, or vice versa, and ship without any automated signal.
Recommendation
Add a test class that exercises validate() against a MessageContainer for each rule in validateSheetData (missing sheet name, missing query, non-SELECT query, trailing semicolon, >20 sheets), and mocks ContentService/Context to verify the create-vs-overwrite branch and the SmartServiceException-wrapping behavior in run().

Medium Priority Findings

Medium Reliability

No validation for duplicate or Excel-illegal sheet names

AdvanceExcelExport.java — validateSheetData(), lines 184–200; ExcelGenerator.java — writeSheet(), line 140

Description
Two or more SQLSheetData entries with the same (or Excel-equivalent, e.g. differing only by trailing space) sheet name, or a name violating Excel's constraints (max 31 characters; cannot contain \ / ? * [ ] :), are never rejected during validate(). They surface only when POI's SXSSFWorkbook.createSheet() throws IllegalArgumentException mid-generation.
Why it matters
Appian's validate() hook exists precisely to give process designers actionable, pre-execution feedback. Letting this fail during run() instead produces a generic POI exception message surfaced through ErrorMessage, which is far less actionable than a targeted validation error naming the offending sheet.
Potential impact
Process designers debug a cryptic runtime failure instead of getting immediate, specific feedback at design/validation time.
Recommendation
In validateSheetData, check sheet-name length (≤31), the illegal-character set, and duplicate names across the full sqlSheetDataList array before execution.
Medium Reliability

No query timeout on per-sheet SQL execution

ExcelGenerator.java — writeSheet(), lines 127–128

Description
PreparedStatement.setFetchSize(FETCH_SIZE) is set, but setQueryTimeout(int) is never called. A slow, blocking, or accidentally unbounded query (e.g. a missing join predicate producing a cross join) can run indefinitely.
Why it matters
Appian Smart Service nodes execute synchronously within a process instance's execution thread; an unbounded query ties up that execution path with no self-imposed ceiling, relying entirely on database-side or Appian-side timeouts (if any) to eventually intervene.
Potential impact
A single misconfigured report query can hang a process instance for an extended period with no clear diagnostic pointing back to this node.
Recommendation
Call statement.setQueryTimeout(...) with a sane default (configurable or fixed, e.g. 120s), and ensure the resulting SQLTimeoutException is surfaced with a clear, sheet-named error message the same way other SQLExceptions already are.
Medium Reliability / Correctness

Editable-column matching is case-sensitive, with no normalization

ExcelExportConfig.java — isEditable(), line 84; ExcelGenerator.java — writeSheet(), line 137

Description
editableColumns.contains(columnName) is an exact, case-sensitive string match against ResultSetMetaData.getColumnLabel(). Column-label casing is JDBC-driver- and database-dependent (e.g. Oracle typically uppercases unquoted identifiers, PostgreSQL lowercases them, SQL Server preserves source casing), and the test suite only exercises this against H2, which happens to return uppercase labels for the queries used.
Why it matters
A process designer who configures editableColumns=["Status"] against a database that returns STATUS or status gets a silently all-locked sheet with no error — matching the documented "no effect" behavior, but in a way that's easy to misconfigure and hard to notice, since the generated file still opens successfully.
Potential impact
Silent, hard-to-diagnose misconfiguration: columns intended to be editable ship locked, and vice versa, without any signal in ErrorMessage.
Recommendation
Normalize case on both sides of the comparison (e.g. compare via equalsIgnoreCase or uppercase both the configured list and the column label at lookup time), and document the chosen behavior explicitly.
Medium Security — Information Disclosure

Raw exception messages are surfaced verbatim to the process-visible error output

AdvanceExcelExport.java — run(), lines 219–223

Description
errorMessage = e.getMessage(); passes whatever message the underlying exception carries — including SQLException messages from the JDBC driver, which frequently embed table/column names, constraint names, or fragments of the failing query — directly into the ErrorMessage output and the thrown SmartServiceException.
Why it matters
Depending on how a calling process model surfaces ErrorMessage (e.g. displayed directly to an end user in a UI), this can leak internal schema details, query fragments, or JNDI/data-source configuration hints to a lower-trust audience than the process designer who wrote the query.
Potential impact
Information disclosure aiding further attack reconnaissance, or simply an unpolished/unprofessional end-user-facing error message.
Recommendation
Keep full exception detail in the SLF4J log (already done via LOG.error) but surface a generic, sheet-scoped message on ErrorMessage (e.g. "Failed to generate sheet 'X' — see server logs for details") unless the process is known to only ever be consumed by trusted designers.

Low Priority & Informational Findings

Low Resource Management

POIFSFileSystem is never closed in encrypt()

ExcelGenerator.java — encrypt(), lines 106–120

Description
fileSystem is constructed, written to via writeFilesystem(out), and never explicitly closed (it implements Closeable). Since this instance holds no external file handle (it's in-memory only), the practical leak risk is low, but it's an inconsistency against the rest of the codebase's otherwise disciplined try-with-resources usage.
Recommendation
Wrap fileSystem in try-with-resources for consistency and to guard against future changes that back it with a real temp file.
Low Correctness

Color.decode has surprising parsing rules for some hex inputs

ExcelGenerator.java — parseHexColor(), lines 252–255

Description
java.awt.Color.decode accepts #, 0x/0X, or a leading 0 as octal — a color string like "099AA1" (no # prefix, leading zero) could parse as an octal literal rather than the intended hex triplet, producing a wrong color instead of the expected NumberFormatException fallback.
Recommendation
Normalize by always prefixing with # and parsing via Integer.parseInt(hex, 16) directly rather than delegating to Color.decode's broader (and here, unwanted) number-format grammar.

Version string drift between pom.xml and appian-plugin.xml

pom.xml line 9 (1.0.0-SNAPSHOT) vs. appian-plugin.xml line 31 (1.0.0)

Description
The Maven artifact version and the plug-in descriptor's declared version are not kept in sync, and nothing enforces that they are.
Recommendation
Either inject the POM version into the descriptor via resource filtering, or add a release checklist step to bump both together.

Informational Technical Debt

Sheet structural protection depends on reflection into a private POI field

ExcelGenerator.java — protectSheet(), lines 294–309

Description
SXSSFSheet doesn't expose the underlying XSSFSheet needed to reach CTSheetProtection, so the code reaches into the private field _sh via reflection. This is already clearly documented as a deliberate, verified workaround, and is protected by a real regression test (testEverySheetIsProtectedWithStructuralEditsAllowed) — but it remains inherently fragile: a future Apache POI upgrade could rename or remove that field with no compile-time warning, only a runtime ExcelGenerationException.
Recommendation
No change needed now given the existing test coverage; when upgrading the poi.version property, re-run the full test suite specifically with this test in mind before treating the upgrade as safe.
Informational Security

JNDI name is a free-text runtime input

AdvanceExcelExport.java — generateExcelBytes(), line 228

Description
context.lookup(jndiName) resolves whatever string is supplied as the jndiName input. This is scoped by the containing application server's JNDI namespace (not an arbitrary remote-codebase lookup as in the historical Log4Shell-style JNDI issues), so the practical risk here is low and is already flagged inline (// nosemgrep comment) as a reviewed, accepted pattern.
Recommendation
Document in the input description that jndiName must only ever reference a pre-configured, admin-approved data source name — never a value derived from end-user input.
Informational Security Hardening

excelPassword is retained as an immutable String

ExcelExportConfig.java, field excelPassword

Description
Java String instances are immutable and not guaranteed to be promptly garbage-collected or zeroed, so the password value can linger in heap memory (and potentially in a heap dump) longer than a char[] that could be explicitly wiped after use.
Recommendation
Low priority given this is a short-lived, single-execution smart service, not a long-running credential store — only worth revisiting if this plug-in is later adapted to hold secrets for longer than one export's lifetime.
Informational DevOps

No CI pipeline automating build, test, or dependency scanning

Repository root — no .github/workflows, no other CI configuration found

Description
mvn test, the OWASP dependency-check-maven scan, and the maven-enforcer-plugin rules are all well-configured but currently only run when a developer remembers to invoke them locally.
Recommendation
Add a CI workflow running mvn -B verify on every push/PR, plus a scheduled (e.g. weekly) job running mvn dependency-check:check given it needs network access to refresh its CVE database and is explicitly excluded from the default build for that reason.

Architecture Review

The plug-in follows a tight, appropriately flat structure for its scope: AdvanceExcelExport (the Appian-facing Smart Service, handling inputs/outputs/validation/orchestration), ExcelExportConfig (an immutable value object encapsulating export settings), ExcelGenerator (the POI-facing rendering engine), SQLSheetData (the plug-in's one custom data type / CDT), and ExcelGenerationException (a narrow unchecked wrapper). This is a reasonable application of separation of concerns: Appian-framework wiring is isolated to AdvanceExcelExport, and none of the POI/Excel-specific logic leaks into it. There is no over-engineering — no unnecessary interfaces, no speculative extension points, no dependency-injection framework beyond what Appian's plug-in loader itself provides via constructor-parameter recognition.

One structural observation: ExcelGenerator at 354 lines carries several responsibilities — SQL execution, cell-value type mapping, style creation/caching, and the reflection-based protection workaround. This is still cohesive (everything is "how to turn a ResultSet into a protected sheet") rather than a true God-class, but if further per-column formatting features are added, splitting cell-value mapping (setCellValue, the JDBC-type switch) into a dedicated small class would keep the file from growing into a genuine maintainability problem.

Concurrency/statefulness: AdvanceExcelExport holds its inputs as mutable instance fields set by Appian via setter methods, which is the SDK's documented lifecycle model. This is safe on the (reasonable, but unverified from code alone) assumption that Appian's process engine instantiates a fresh Smart Service instance per node execution rather than pooling and reusing one instance across concurrent executions; the code itself gives no explicit guarantee either way. Everything downstream (ExcelExportConfig, ExcelGenerator) is either immutable or locally-scoped per call, so no additional statefulness risk is introduced beyond what the SDK contract itself implies.

Security Review

Authentication and authorization are entirely delegated to the Appian platform, which is the correct model for a Smart Service plug-in — there is no bespoke auth logic to review here, and none should exist. The security surface specific to this plug-in is: the SQL text it executes, the JNDI name it resolves, the error detail it exposes, and the password/encryption path it offers.

Sheet-level cell protection deserves a clarifying note rather than a finding: protectSheet() applies protection with an empty password by design (documented in-code), meaning any Excel user can remove it via the standard "Unprotect Sheet" UI with no real password challenge. Performance Review

The core generation path is well-engineered for its purpose. SXSSFWorkbook with a 100-row access window bounds heap usage independent of total row count, matched by PreparedStatement.setFetchSize(100) on the JDBC side so result rows are pulled from the database in batches rather than materialized all at once. The 50,000-row test (testLargeDataset) both documents and verifies this claim rather than leaving it as an assumption, and additionally asserts a wall-clock ceiling (<60s), giving the suite an actual performance regression guard, not just a functional one.

The pre-cached DataStyles (6 style variants reused across every cell) avoids a well-known and easy-to-hit Apache POI failure mode — creating a new CellStyle per cell can exhaust the workbook's finite style-slot budget on datasets with more than a few thousand rows. This is a detail many hand-rolled Excel exporters get wrong, and it's handled correctly here.

One efficiency nit, informational-severity only: ExcelExportConfig.isEditable() does an O(n) List.contains scan per column, per row-style lookup call site (though in practice it's only called once per column during header setup — editableFlags is computed once and reused for all data rows, so this is not actually a hot-path cost). No change needed.

Scalability Review

Horizontal scaling, load balancing, and distributed-system concerns are out of scope for this artifact by design: an Appian Smart Service plug-in is a synchronous unit of work executed inline within a process instance by Appian's own process engine, which already owns scaling and concurrency at the platform level. There is no service to scale independently, no request queue to size, and no shared mutable state across executions to reason about beyond what's covered in the Architecture section above.

The one scalability dimension genuinely under this plug-in's control — handling arbitrarily large result sets without proportional memory growth — is addressed correctly via the streaming architecture discussed under Performance. The MAX_SHEETS = 20 cap is a sensible, explicit bound against unbounded workbook fan-out; there is no equivalent cap on rows-per-sheet, but given the streaming design that's a reasonable choice rather than an oversight — the limiting factor for very large sheets is generation wall-clock time and Excel's own ~1,048,576-row ceiling, not plug-in memory.

Maintainability Review

Naming is consistent and descriptive throughout (writeHeaderRow, writeDataRows, findExistingDocumentId read as exactly what they do). Comments are used correctly by this codebase's own standard — they explain why a non-obvious choice was made (the POI relocation exception, the reflection workaround, the deprecated-API usage) rather than restating what the next line does. This is a genuinely good example of comment discipline and should be treated as the standard to hold future changes to, not relaxed.

The Builder pattern in ExcelExportConfig is applied correctly and proportionately — seven fields, three of them required — this is right at the size where a builder earns its keep over a telescoping constructor, without being overkill. No dead code, no duplicated logic, and no unused fields or methods were found in the reviewed files.

Testing Review

ExcelExportConfigTest (6 tests) covers the builder's validation and default-value substitution logic completely — every branch in the constructor and required-field check has a corresponding assertion. ExcelGeneratorTest (10 tests) is the stronger of the two suites: it verifies structural protection flags directly against the underlying CTSheetProtection XML model (not just POI's higher-level API), verifies encryption via an actual password-verify-and-decrypt round trip rather than only checking the file "looks encrypted," and includes both a multi-sheet cross-shape test and the large-dataset performance-adjacent test discussed above.

The gap, covered as a High finding above, is AdvanceExcelExport itself. Beyond adding coverage for validate() and the create/overwrite branch, a mocked-DataSource test of generateExcelBytes()'s JNDI-lookup-failure path (what happens when context.lookup() throws or returns something that isn't a DataSource) would also close a real edge case: today, a ClassCastException from the unchecked cast on line 228 would propagate as a generic, unhelpful error message rather than a targeted "JNDI name did not resolve to a data source" message.

Observability Review

Logging is present at appropriate levels throughout the generation path (debug for per-sheet start and JNDI lookup, info for completion, warn for the recoverable invalid-header-color fallback, error for terminal failures) via SLF4J, correctly left at provided scope so Appian's own logging backend and routing apply. There are no metrics, distributed tracing, or health-check endpoints — this is appropriate for a synchronous, in-process plug-in with no independent runtime to monitor; Appian's own process-instance audit trail and admin console already provide execution-level visibility for this kind of node. No changes recommended here beyond what's already in place.

DevOps Review

The Maven build is well-configured for an Appian plug-in specifically: maven-shade-plugin relocates POI to avoid classpath conflicts with Appian's own bundled version (with a documented, specific reason for the one exception), release>11</release> on the compiler plugin correctly gates against the real Java 11 API surface even when building on a newer JDK, and the enforcer/versions/dependency-check plugins are all present and correctly scoped (network-dependent CVE scanning deliberately kept out of the default build lifecycle). The one gap is automation: none of this — build, test, or scan — currently runs anywhere other than a developer's own machine. See the DevOps informational finding above for the specific recommendation.

Recommended Refactoring Opportunities

  • Extract cell-value mapping. Move setCellValue()'s JDBC-Types switch out of ExcelGenerator into a small dedicated JdbcCellValueWriter-style helper if any further per-type formatting (e.g. currency, percentage) is added — keeps ExcelGenerator focused on sheet/style orchestration.
  • Centralize sheet-name validation. A single SheetNameValidator (or a static method) used both by AdvanceExcelExport.validate() and, defensively, by ExcelGenerator.writeSheet() would close the Medium finding above without duplicating the Excel naming-rule constants in two places.
  • Introduce a narrow SQL-shape check. A small utility that rejects embedded semicolons anywhere (not just a trailing one) and multiple top-level SELECT keywords would meaningfully raise the bar over the current prefix/suffix check without requiring a full SQL parser dependency.

Quick Wins

  • Normalize editableColumns/column-label comparison to be case-insensitive (single-line change in ExcelExportConfig.isEditable).
  • Wrap the POIFSFileSystem in encrypt() with try-with-resources.
  • Replace Color.decode with a direct Integer.parseInt(hex, 16) parse to avoid octal-literal surprises.
  • Add statement.setQueryTimeout(...) in ExcelGenerator.writeSheet().
  • Correct the README's stated test count (or de-specify it).
  • Reject embedded semicolons anywhere in the SQL text, not only a trailing one.

Long-Term Recommendations

  • Build a unit test suite for AdvanceExcelExport covering validate(), the document create/overwrite branch, and exception-to-output mapping, using mocked ContentService/Context instances.
  • Stand up a CI workflow (build + test on every change; scheduled dependency-check run) so the already-well-configured Maven quality gates actually execute automatically rather than depending on developer discipline.
  • Once the plug-in has real deployment history, add an explicit <application-version min="..."/> to appian-plugin.xml once the minimum supported Appian version is confirmed (already flagged as a known, deliberate gap in the descriptor's own comments).
  • If usage broadens beyond trusted process designers authoring the SQL text directly, revisit the injection-hardening recommendations under Security before that trust boundary changes.

Overall Assessment

This is a small, focused, and genuinely well-executed plug-in for the problem it solves. Its architecture is proportionate to its scope, its streaming Excel generation is correctly engineered and tested against a realistic large-dataset scenario, and its documentation is meaningfully above the bar typically seen in projects of this size — it captures specific, verified production failure modes rather than generic advice. The gaps identified are concentrated and addressable: strengthen SQL input validation and its documentation, close the test-coverage hole around the Appian-facing orchestration class, and tighten a handful of edge cases (case-sensitive column matching, sheet-name validation, query timeouts). None of them are architectural — they're all additive hardening on top of a sound foundation, achievable without restructuring anything.

Final Recommendation

APPROVE WITH COMMENTS

Safe to deploy as-is for use by trusted process designers. Address the two High findings (SQL validation documentation/hardening, orchestration-class test coverage) before broadening adoption to less-trusted callers or process models that might build query text from dynamic input.

All findings are grounded directly in this source; no external dependency source (Apache POI, Appian SDK internals) was inspected beyond what the plug-in's own code and comments reference.
